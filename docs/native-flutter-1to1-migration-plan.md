# DUNES 原型转原生 Flutter 1:1 还原计划

## 最终确认（强约束）

- **后端接口沿用现有**：不新增接口、不改接口路径、不改请求/响应字段、不改 WS 事件协议。
- **前端仅做原生化重写**：把现有 HTML/WebView 功能迁移为 Flutter 原生实现，不改变业务语义。
- **UI 风格保持当前版本**：色彩、排版、层级、文案、图标、交互路径按现有前端 1:1 还原。
- **任何偏离必须评审**：若出现必须偏离上述约束的情况，需先评审并记录回滚方案。

## 结论

可行，且建议尽快执行。  
当前「HTML + WebView 注入」方案在 Android/iOS 上会持续遇到能力边界（文件选择、相机/麦克风权限、录音、输入法、系统兼容性、性能），长期维护成本会越来越高。  
目标应改为：**HTML 仅作视觉与交互参考，业务与 UI 全部原生 Flutter 实现**。

## 总目标

- 1:1 还原现有可用功能与交互（视觉误差可控）
- Android / iOS 同步稳定发布
- 逐步下线 WebView IM/审批/Nova 核心流程
- 建立可持续迭代的 Flutter 架构（状态、网络、实时、媒体、埋点、测试）

## 迁移原则

- **先核心后边缘**：先迁移登录、工作台、通讯、聊天，再迁移长尾页面
- **双轨并行**：新原生页上线前，旧 WebView 页保留兜底入口
- **小步快跑**：每周可交付、可灰度、可回滚
- **先协议后界面**：先稳定 API/WS/媒体上传协议，再做精细 UI 还原
- **单一数据源**：严禁同功能同时维护两套状态逻辑

## 零后端改动 / 零 UI 改版约束

### 不可变更清单（硬约束）

- **后端服务不改**：不新增、不重构、不替换现有服务
- **接口协议不改**：现有 API 路径、请求参数、响应字段、错误码保持不变
- **实时协议不改**：现有 WS 地址、频道命名、事件结构保持不变
- **业务规则不改**：审批流转、会话排序、未读策略、权限判定保持不变
- **页面视觉不改**：布局结构、色值、字号层级、文案、图标语义保持不变

### 允许的最小差异（白名单）

- 系统字体渲染差异（1~2px 级）
- 系统手势/输入法行为差异（返回手势、键盘弹出时机）
- 动画曲线微差（不改变交互意图与时长级别）
- 个别系统组件外观差异（系统权限弹窗、系统文件选择器）

### 研发执行红线

- 不以「改后端」规避 Flutter 端问题
- 不以「改 UI 设计」降低还原成本
- 不在同一功能长期维护两套业务实现（WebView 与 Native）
- 所有偏离必须先过评审并记录（原因、范围、回滚方案）

### 验收标准（针对 1:1 约束）

- **接口兼容率 100%**：抓包对比请求路径、字段、鉴权头一致
- **业务一致率 100%**：同账号同操作，关键结果一致
- **视觉一致率 >= 95%**：按页面清单逐项比对（布局/间距/字号/颜色/状态）
- **交互一致率 >= 95%**：点击路径、反馈、边界态处理一致
- **问题处理闭环**：任何偏差都要有 issue、归因、修复和回归记录

## 当前已完成前端能力基线（必须 100% 保留）

## 迁移对象拆分（按功能域）

### 模块 1：鉴权与会话

- 登录、token 刷新、退出登录、会话恢复
- 产出：`auth` 模块 + 会话状态仓库 + 错误码映射

### 模块 2：通讯列表与联系人

- 会话列表、联系人、搜索、未读、在线态
- 产出：`conversation`/`contact` 模块 + realtime 事件分发器

### 模块 3：私聊/群聊核心

- 消息列表、分页、已读回执、撤回/更新、滚动定位
- 产出：`chat_core` 模块 + message store + send pipeline

### 模块 4：媒体与输入能力

- 拍照、相册、文件、语音、截图、粘贴拖拽
- 产出：`chat_media` 模块 + 权限管理 + 上传任务管理器

### 模块 5：审批与 Nova

- 审批主链路与 Nova 核心会话能力
- 产出：`approval` / `nova` 模块 + 统一历史与草稿恢复

## 每个功能项的统一 DoD（完成定义）

- **接口一致**：请求 URL、method、headers、payload、响应解析完全对齐
- **状态一致**：列表刷新、未读变化、已读回执、失败重试行为一致
- **交互一致**：点击路径、反馈文案、禁用态、边界态一致
- **视觉一致**：关键区域截图对比通过（误差在白名单范围内）
- **回归通过**：Android + iOS 真机各至少一台通过用例
- **可观测性**：失败可定位（日志、错误码、关键事件）

## 迁移过程的发布与回滚策略

- 每个模块独立开关（remote config）
- 新模块先灰度给内部账号，再扩大比例
- 任一 P0 回归失败立即回切 WebView 路径
- 每次发布必须附带“新增能力 + 风险点 + 回滚步骤”

## 建议架构（原生）

- 状态管理：`Riverpod`（或 Bloc，二选一，建议 Riverpod）
- 网络层：`dio` + 统一拦截器 + 错误模型
- 实时通信：`centrifuge-dart`（或 websocket + 自定义协议层）
- 文件/媒体：
  - 图片拍照/相册：`image_picker`
  - 文件选择：`file_picker`
  - 录音：`record`
  - 音频播放：`just_audio`
- 本地存储：`shared_preferences` + `isar/hive`（消息缓存可选）
- 导航：`go_router`
- 设计系统：沉淀 `tokens + widgets`（按钮、输入框、消息气泡、列表项）

## 分阶段计划（建议 8~12 周）

### Phase 0：基线与冻结（3~5 天）

- 冻结 WebView 新功能，仅修阻塞 bug
- 产出完整功能清单（按 screenId：`B2/C1/C2/C4/C5/...`）
- 确认 API 契约、实时事件模型、媒体上传流程
- 定义验收标准（功能 + 性能 + 崩溃率 + 可用性）

### Phase 1：基础设施（1~2 周）

- 搭建原生项目骨架（路由、状态、主题、网络、日志、错误上报）
- 登录态、鉴权、用户资料、全局配置完成
- 统一权限申请流程（相机/麦克风/相册/通知）
- 搭建 UI 组件库基础（AppBar、列表、卡片、输入区、弹窗）

### Phase 2：工作台与通讯列表（1~2 周）

- 原生还原 `B2` 工作台
- 原生还原会话列表/联系人列表（`C1` 等）
- 完成未读、在线态、排序、搜索
- 接通 realtime 事件驱动列表刷新

### Phase 3：聊天核心（2~3 周）

- 原生还原私聊/群聊（`C5/C2`）
- 文本、图片、文件、语音消息完整打通
- 输入区工具栏：拍照、相册、文件、语音、表情
- 消息回执/已读、撤回、分页加载、滚动定位
- 这阶段完成后即可替换掉 WebView IM 主路径

### Phase 4：审批与 Nova 核心（2~3 周）

- 审批主流程页（发起、列表、详情）原生化
- Nova 聊天核心能力迁移（消息流、代码块、附件）
- 历史会话、状态同步、草稿恢复
- 保留复杂次要页暂时 WebView 兜底

### Phase 5：收尾与下线（1~2 周）

- 全量回归、性能优化、埋点补齐
- 灰度发布（10% → 30% → 100%）
- 移除 WebView 相关注入脚本与桥接代码
- 文档完善：开发规范、组件规范、发布流程

## 功能迁移优先级（P0/P1/P2）

- **P0（必须先做）**：登录、会话列表、私聊/群聊、发送图片/文件/语音、实时推送、未读与已读
- **P1（第二梯队）**：审批关键链路、Nova 核心对话、资料与联系人管理
- **P2（尾项）**：低频页面、复杂报表页、历史兼容入口

## 风险与应对

- 协议变动频繁：先锁 API 版本，变更走评审
- UI 还原成本高：建立组件库，减少页面级重复实现
- 迁移周期拉长：按 P0/P1/P2 分层，优先保证核心可发布
- 双轨期数据不一致：统一仓库层，WebView 只读或逐步下线写路径

## 验收标准（每阶段）

- 功能通过率 >= 95%
- P0 用例全部通过
- Android/iOS 真机主流程无阻塞 bug
- 崩溃率、ANR、卡顿指标达到上线阈值
- 关键链路日志可追踪（登录、发消息、上传、实时连接）
