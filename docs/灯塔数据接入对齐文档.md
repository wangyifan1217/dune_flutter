# 灯塔移动端 — 数据接入对齐文档

> **目的**：把 dart 端 (`native_lighthouse_page.dart`) **读到的字段** ↔ **写死的占位** ↔ **v8 脚本派生口径** ↔ **资管库实表列** 一次性对齐到位，关闭所有「数字看起来不准」「切了 period 数字没变」「detail 页永远是月」「woa/SAAS/discount 是 0」这类问题。
>
> **依据**：`build_l2_v8.py`（v8 派生权威口径）、`二级页面总览指标版_v8.xlsx`（28 sheet 验证基线）、`lighthouse-database-fields.md`（宽表盘点）。
>
> **范围**：仅讨论灯塔一级三 tab + 二级详情页 + 趋势卡 共享的数据接入。不动 UI 交互、不动 v8 业务规则。

---

## 0. TL;DR — 一张表看完所有缺口

| # | 问题 | 现状 | 根因 | 修法（P0/P1） |
|---|------|------|------|---------------|
| ① | 切日/周/季/年时 hero 数字不真实 | 前端 `_kPeriodMult = {1/30, 7/30, 1.0, 3.0, 12.0}` 把月数据乘出来 | 后端只返月聚合 | **P0** 后端按 `period` 查 daily / monthly / yearly 三表 |
| ② | hero 上的「603.2万 / +5.2% / 中石油 64%」等数字 | `_kPeriods` 5 套常量硬编码 | 后端没返这些 | **P0** 后端 `metrics` 字段返回真聚合 |
| ③ | 点进二级页数字不随 period 变 | `_buildDetailView` 直接读 `entity[...]`，无 mult | 详情字典只有月版 | **P0** detail 也按 period 服务端聚合 |
| ④ | `name/group` 找不到字段 | 宽表 `category_l1_name=成品油零售`、`merchant_name=广东中石油`、`channel_category_*=NULL` | 缺 v8 派生列 | **P0** sync 写入 `tag1/tag2/tag3/supply_class/channel_class/product_line_group/oil_type` |
| ⑤ | woa 永远 0 | 宽表无列 | 数据源未定义 | **P1** 宽表 ADD `woa` 列；先按 0 返回但口径需业务方给 |
| ⑥ | saasFee 全 0 | 宽表 `saas_service_fee` 全 0（ingest_line 没匹配上 mapping） | `field_mapping` 配置不全 | **P0** 排查 ingest_line → mapping → 宽表，补齐 |
| ⑦ | discount 进度（供给方折扣） | 前端展示但 API 不返 | 不在宽表 | **P1** API 查 `light_tower_sp_disrules*` 4 张表算进度 |
| ⑧ | 渠道详情 key `name::group` 拼不出 | 渠道分类未派生 | 同 ④ | **P0**（同 ④ 一并解决） |

> 完成 P0 后，前端可以删 `_kPeriods` / `_kPeriodMult` 两个常量；hero、list、detail、趋势卡全部读后端返的真聚合。

## 1. 现状审计：dart 究竟读了什么

### 1.1 一级 tab 列表（`_bundle.rowsOf(tab)` 每行）

| 字段 | 类型 | 现读法 | 备注 |
|------|------|--------|------|
| `name` | string | `r['name']` | 一级行名（dart `_buildListItem` 第 2017 行） |
| `group` | string | `r['group']` | 用于上色 + 详情 key（`name::group`） |
| `profit` | num | `r['profit'] * mult` | 默认排序键 |
| `sales` | num | `r['sales'] * mult` | 毛利率分母 |
| `gmv` | num | `r['gmv'] * mult` | 引流 GMV |
| `revenue` | num? | `(r['revenue'] ?? r['sales']) * mult` | 缺值时回落 sales |
| `cost` | num | `r['cost'] * mult` | 业务成本 |
| `tax` | num | `r['tax'] * mult` | 税务成本 |
| `spread` | num | `r['spread'] * mult` | 利差 |
| `woa` | num | `r['woa'] * mult` | 现全 0 |
| **product only** | | | |
| `gmv2` | num | 同上 | GMV（与 sales 同源） |
| `its` | num | 同上 | 核销金额 |
| `itsAfter` | num | 同上 | 折后 ITS（= sales） |
| `totalCost` | num | 同上 | 含项目成本的总成本 |
| **supply / channel only** | | | |
| `saasFee` | num | 同上 | SAAS 服务费 |
| `projectCost` | num | 同上 | 项目成本（v8 现占位 0） |
| `deferred` | num | 同上 | 抵扣延期分润（v8 现占位 0） |
| `discount` | num | 同上 | 折扣进度（supply only，前端展示） |

→ 后端必须按 v8 命名返每行这套字段（camelCase）。

### 1.2 详情字典（`_bundle.productDetail[key]` / `supplyDetail` / `channelDetail`）

```jsonc
{
  "<entity-key>": {
    "supply":      [ { /* 同 1.1 行结构 */ }, ... ],  // 仅 product 有
    "channel":     [ ... ],   // 仅 product / supply
    "product":     [ ... ],   // 仅 supply / channel
    "project":     [ ... ],   // 所有
    "productName": [ ... ]    // 所有，SKU 级
  }
}
```

- `product_detail` 的 key = `标签一`
- `supply_detail`  的 key = `标签二`
- `channel_detail` 的 key = `标签三::渠道分类`（如 `产险::平安`）

### 1.3 dart 当前**写死**的东西（必须迁到后端）

| 常量 | 行号 | 内容 | 后端要怎么给 |
|------|------|------|--------------|
| `_kPeriods` × 5 | 57–113 | 每个 period 的 `label / salesV / salesU / profitV / gmvV / rate / deltaDir / deltaVal / deltaVs / compose[]` | 接口顶层 `metrics` 对象按当前 period 返一组真数 |
| `_kPeriodMult` | 117 | `day=1/30, week=7/30, month=1, quarter=3, year=12` | **删掉**。每次切 period 重新 `fetchOverview(period:k)`，rows 已经是该 period 的真聚合 |
| `_kPeriodVs` | 125 | `vs 昨日 / vs 上周 / ...` | 可以留前端，也可以由 `metrics.deltaVs` 返回 |
| `_kPeriods[*].compose` | 62、73、84、95、106 | hero 下方 4 段占比条 | 后端 `metrics.compose: [{name, pct, colorKey}]` |

### 1.4 dart 当前**半接入**的地方

- **二级详情**（`_buildDetailView`，第 2432 行）：`final entityProfit = (entity['profit'] as num?)?.toDouble() ?? 0;` —— **没有 mult**。结果是切 period 时一级页变了、点进去二级页永远还是月数。要么前端补 mult（仍然是伪聚合），要么按 P0 走后端 period-specific。本文档主张后者。

## 2. v8 → API → dart 三层映射

### 2.1 维度字段（这是数据接入最根本的卡点）

| dart `name` | dart `group` | v8 派生列 | 后端聚合 GROUP BY |
|-------------|--------------|-----------|-------------------|
| product tab | | | |
| `r['name']` | `r['group']` | `标签一` / `product_line_group` | `GROUP BY tag1, product_line_group` |
| supply tab | | | |
| `r['name']` | `r['group']` | `标签二` / `供给分类` | `GROUP BY tag2, supply_class, oil_type` |
| channel tab | | | |
| `r['name']` | `r['group']` | `标签三` / `渠道分类` | `GROUP BY tag3, channel_class` |

**v8 派生规则（必须 1:1 还原到后端 sync 阶段）：**

| 派生列 | v8 函数 | 入参 | 现状 |
|--------|---------|------|------|
| `tag1` | `relabel_summary_v3.classify()` | 旧标签、产品名、数据来源、折扣模式 | ❌ 需移植；ingest.`tag_l1` 全 NULL，不能直接用 `category_l1_name`（"成品油零售"） |
| `tag2` | `COALESCE(省份_中石油, 省份)` | `province_name` + 「省份_中石油」 | ⚠️ 省份_中石油列未灌；need raw_payload 解析 |
| `supply_class` | `map_supply_class(prov, prov_cnpc)` | 同上 | 派生即可 |
| `oil_type` | `map_oil(supply_class, product_name)` | + `am_channel_product.oil_category` | 派生即可（主数据有 oil_category） |
| `channel_class` | `map_channel_class(客户签约主体, 标签一, 项目名称)` | 缺 **客户签约主体**（宽表无） | ❌ 需从 ingest_line.`counterparty_entity` 或 raw_payload 解析 |

## 3. API 契约（建议固化）

### 3.1 `GET /lighthouse/overview?period={day|week|month|quarter|year}[&date=YYYY-MM-DD]`

`date` 不传默认今天 / 本周 / 本月 / 本季 / 今年（按 period 解析）。

### 3.2 Period → 数据源

| period | 表 | WHERE | 备注 |
|--------|-----|-------|------|
| `day` | `am_lighthouse_operating_daily` | `stat_date = :date` | 单日 |
| `week` | `am_lighthouse_operating_daily` | `stat_date BETWEEN :weekStart AND :weekEnd` (周一–周日) | 7 日 SUM |
| `month` | `am_lighthouse_operating_monthly` | `stat_month = :month` | 单月 |
| `quarter` | `am_lighthouse_operating_monthly` | `stat_month IN (:m1,:m2,:m3)` | 3 月 SUM |
| `year` | `am_lighthouse_operating_yearly` | `stat_year = :year` | 年聚合 |

`week` 与 `quarter` 不要新建表，直接在 daily/monthly 上 SUM。

**关键约束：**
- 三个 tab 的一级行结构与对应 detail sub-list 行结构**完全一致**（同样的 camelCase 指标键），前端用一套 `_buildListItem` 渲染。

- `compose.colorKey` 只能是 dart 端 `lhGroupColor()` 认识的 7 个值：`cnpc / sinopec / private / carrier / pingan / dict / multi / unk`。
- 渠道 detail 的 key 必须是 `${tag3}::${channel_class}`，前端按 `key.split('::')` 解析。

### 3.4 接口数量与缓存

- 切 period 时前端 **重新调用一次** `fetchOverview(period)`。后端 Redis 缓存这一份就够（TTL 30s ~ 5min 看实时性需求）。
- detail 字典随 overview 一起返；不再做"列表先回 + detail 懒加载"，省一次往返、避免一致性问题。
- 实测当前数据量（daily 3.1k / monthly 174 / yearly 174 行）单次 payload < 200KB，可接受。

## 4. 前端改动清单（P0 接入完成后）

### 5.3 P0 — 一级聚合 SQL（直接对应 v8 Sheet 1/2/3）

```sql
-- 产品 tab 一级（对齐 v8 Sheet 1）
SELECT tag1 AS name,
       product_line_group AS `group`,
       SUM(profit)              AS profit,
       SUM(sales_amount)        AS sales,
       SUM(gmv)                 AS gmv,
       SUM(sales_amount)        AS gmv2,        -- v8: 同 sales
       SUM(verify_amount)       AS its,
       SUM(sales_amount)        AS itsAfter,   -- v8: 同 sales
       SUM(spread_margin)       AS spread,
       SUM(woa)                 AS woa,
       SUM(revenue)             AS revenue,
       SUM(business_cost)       AS cost,
       SUM(tax_cost)            AS tax,
       SUM(total_cost)          AS totalCost
FROM am_lighthouse_operating_daily
WHERE is_del = 0 AND stat_date BETWEEN :start AND :end
  AND tag1 IS NOT NULL
GROUP BY tag1, product_line_group
ORDER BY profit DESC;

-- 供给 tab 一级（对齐 v8 Sheet 2）
SELECT tag2 AS name,
       supply_class AS `group`,
       SUM(profit) AS profit, SUM(sales_amount) AS sales, SUM(gmv) AS gmv,
       SUM(business_cost) AS cost, SUM(tax_cost) AS tax, SUM(spread_margin) AS spread,
       SUM(saas_service_fee) AS saasFee, SUM(project_cost) AS projectCost,

FROM am_lighthouse_operating_daily
WHERE is_del = 0 AND stat_date BETWEEN :start AND :end AND tag2 IS NOT NULL
GROUP BY tag2, supply_class, oil_type
ORDER BY profit DESC;

-- 渠道 tab 一级（对齐 v8 Sheet 3）
-- name = tag3, group = channel_class
-- detail key = CONCAT(tag3, '::', channel_class)
SELECT tag3 AS name,
       channel_class AS `group`,
       SUM(profit) AS profit, SUM(sales_amount) AS sales, SUM(gmv) AS gmv,
       SUM(business_cost) AS cost, SUM(tax_cost) AS tax, SUM(spread_margin) AS spread,
       SUM(saas_service_fee) AS saasFee, SUM(project_cost) AS projectCost,
       SUM(deduct_deferred_profit_share) AS deferred, SUM(woa) AS woa
FROM am_lighthouse_operating_daily
WHERE is_del = 0 AND stat_date BETWEEN :start AND :end AND tag3 IS NOT NULL
GROUP BY tag3, channel_class
ORDER BY profit DESC;
```

monthly/yearly 把表名和 `WHERE` 时间键换掉即可。

### 5.4 P0 — `metrics` 顶层对象

```sql
-- 当前 period 全量聚合
WITH cur AS (
  SELECT SUM(sales_amount) sales, SUM(profit) profit, SUM(gmv) gmv
  FROM am_lighthouse_operating_daily
  WHERE is_del = 0 AND stat_date BETWEEN :start AND :end
),
prev AS (   -- 上一个对照期（昨日/上周/上月/...）
  SELECT SUM(sales_amount) sales
  FROM am_lighthouse_operating_daily
  WHERE is_del = 0 AND stat_date BETWEEN :prevStart AND :prevEnd
),
compose AS (   -- hero 下方占比条：按 channel_class 取 top-3 + 其他
  SELECT channel_class, SUM(sales_amount) s
  FROM am_lighthouse_operating_daily
  WHERE is_del = 0 AND stat_date BETWEEN :start AND :end
  GROUP BY channel_class ORDER BY s DESC
)
SELECT cur.sales, cur.profit, cur.gmv,
       (cur.sales - prev.sales) / NULLIF(prev.sales, 0) * 100 AS deltaPct,
       (cur.sales >= prev.sales) AS deltaUp
FROM cur, prev;
```

deltaPct 正→`deltaDir=up`，负→`down`。`compose` 取 top-3，其余汇总成"其他"。
